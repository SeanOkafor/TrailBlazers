- Java JDK 21 (or compatible version)

```powershell
cd src
javac util/*.java

cd ..
```

```powershell
java -classpath src MainWindow
```

## Bug Fix: Tutorial Enemy Hitbox Too Large (March 1, 2026)

### Problem
The Tutorial Enemy's collision hitbox used the full 395×395 display area, but the actual
sprite content (the block) only fills roughly the centre ~195×195 pixels. This caused
projectiles to register hits in the transparent padding surrounding the sprite.

### Fix (TutorialEnemy.java)
- Added `COLLISION_INSET_X` and `COLLISION_INSET_Y` constants (100 px each) that define
  the transparent padding on each side of the sprite image.
- Updated `collidesWith()` to compute a tighter hitbox:
  ```
  hitX = x + COLLISION_INSET_X
  hitY = y + COLLISION_INSET_Y
  hitW = DISPLAY_WIDTH  - 2 * COLLISION_INSET_X   (= 195)
  hitH = DISPLAY_HEIGHT - 2 * COLLISION_INSET_Y   (= 195)
  ```
- AABB check now uses the inset rectangle instead of the full display area.

## Final Boss (Level 2) — 3-Phase Boss (March 2–3, 2026)

### Overview
The Final Boss appears only in Level 2. It has **3 phases**, each with **25,000 HP**
(total health bar: **75,000 HP**). Each phase is essentially a separate character with
its own animated sprites, sharing a single persistent health bar.
Constructor: `FinalBoss(boolean multiplayer, Player1 player1, Player2 player2)`

### Sprites
Each phase has 3 looping animation frames (stored in separate subfolders):
- `res/New Graphics/Final Boss/phase1/FBphase1-{1,2,3}.png`
- `res/New Graphics/Final Boss/phase2/FBphase2-{1,2,3}.png`
- `res/New Graphics/Final Boss/phase3/FBphase3-{1,2,3}.png`

Attack sprites:
- `res/New Graphics/Final Boss/phase1/Attack1/sprite-1-{1,8,12}.png` (157×60 → 150×58)

### State Machine
The boss cycles through 7 states (same pattern as FirstBoss):
| State | Description |
|-------|-------------|
| **ENTERING** | Descends from above the screen to the centre (3 px/frame) |
| **IDLE** | At home position, 0.6s pause, then RNG picks next attack |
| **ATTACKING** | Performing an attack pattern (see Phase 1 Attacks below) |
| **RETURNING** | Moving back to home position after attack finishes (4 px/frame) |
| **SLIDING** | Phase HP depleted → slides off right edge (5 px/frame) |
| **WAITING** | Off screen, 60-frame pause before next phase enters |
| **DEFEATED** | All 3 phases beaten, boss removed |

### Phase Transitions
1. Player deals 25,000 damage → current phase HP hits 0
2. Health bar **locks** (no more damage during transition)
3. Current phase slides off the right edge of the screen
4. Brief pause (~0.6 s)
5. Next phase descends from the top into position
6. Boss becomes IDLE again — health bar unlocks

### Phase 1 Attacks (March 3, 2026)
When IDLE, the boss randomly selects Attack 1 or Attack 2 via `Random.nextInt(2)`.
After the attack completes, the boss returns to its home position and picks again.
Boss can be damaged during IDLE, ATTACKING, and RETURNING states.

#### Phase 1 Attack 1 — Fireball Barrage
- Boss bobs vertically between Y=100 and Y=700 at 3 px/frame
- Fires a 3-frame animated fireball every 72 frames (20% faster than First Boss P2A2's 90)
- Fireballs spawn from the left edge of the boss, move left at speed 9
- Fireball display size: 150×58 (from 157×60 native sprites)
- Attack lasts 7.2 seconds (720 frames → 10 fireballs), then transitions to RETURNING
- Deals 1 heart of damage on player contact + triggers i-frames

#### Phase 1 Attack 2 — Charge/Ram
- Boss slides off the right edge at speed 10
- While off-screen, teleports to a random Y on the right edge
- Flies horizontally left at speed 12 (faster than FirstBoss for difficulty)
- If the boss body overlaps a player (AABB with collision insets), deals **2 hearts**
  of damage using `player.hit(int amount)`
- Players can still shoot the boss during the charge
- When the boss goes off the left edge, it teleports to a new random Y on the right
- Repeats 3 times, then transitions to RETURNING
- Sub-states: SLIDE_OFF → CHARGE_LEFT

### Attack Inheritance (March 3, 2026)
Each phase inherits ALL attacks from previous phases and adds new ones:
- **Phase 1:** Attack 0 (Fireball Barrage), Attack 1 (Charge/Ram) — 2 attacks
- **Phase 2:** Inherits Phase 1 attacks + Attack 2 (Zip & Shoot) — 3 attacks
- **Phase 3:** Inherits all previous + Attack 3 (Mini Boss) — 4 attacks

Attack is chosen via `rng.nextInt(numAttacks)` from the full pool available to
the current phase.

#### Phase 2 Attack 2 — Zip & Shoot
- Boss zips at speed 25 to a random position on the right half of the screen
- Pauses for 0.25 seconds (25 frames)
- Fires **3 big fireballs stacked vertically** (centre + 80px above + 80px below)
  from the boss's left edge, flying left at speed 14
- Fireballs reuse the Phase 1 fireball sprites but displayed larger (200×77)
- Zips to a new random position and repeats — **4 times total**
- After all 4 volleys, transitions to RETURNING
- Sub-states: ZIP_TO_POS → PAUSE → SHOOT (loop)

#### Phase 3 Attack 3 — Mini Boss Summon
- Boss stays at home position; spawns a **mini boss** slightly right of centre (x=550)
- Mini boss sprites: `phase3/Attack1/sprite-1-{1,4,6}.png` (175×196, 3-frame loop)
- Mini boss has **5000 HP**, stays perfectly still
- Fires **homing bolts** (same pattern as FirstBoss P2A1):
  - Spawns bolts one at a time every 0.3s (30 frames) at the mini boss's centre
  - Each bolt sits still for 0.5s (50 frames), then aims at a random player and launches at speed 6
  - 5 bolts per wave, continuous waves until mini boss is defeated
  - Bolts reuse the Phase 1 fireball sprites at 84×42 display size
- Mini boss has its own health bar (120×10, green/yellow/red) above its sprite
- Mini boss has red damage flash (same technique as main boss)
- Mini boss hitbox: AABB with 20px inset on each side
- Player projectiles can hit the mini boss (collision check in Level2Panel)
- Attack lasts **until the mini boss is defeated** (HP reaches 0)
- After defeat: 2-second delay (200 frames) before boss transitions to RETURNING
- Mini boss is cleaned up on phase transitions (boss phase HP reaching 0)

### Health Bar
- 350×24 bar above the boss sprite, showing total HP / 75,000
- Colour: green (>50%) → yellow (20-50%) → red (<20%)
- "Phase X / 3" label drawn above the bar

### Damage Flash
Same SRC_IN red-tint overlay technique as the Tutorial Enemy (45% alpha, 10 frames).

### Collision
- Same inset AABB approach as Tutorial Enemy (75 px inset on each side)
- Registers hits when boss is in IDLE, ATTACKING, or RETURNING state
- Projectile collision updates firing player's damage dealt + special charge

### Enemy Projectiles in Level 2
- `FinalBoss.getEnemyProjectiles()` returns the boss's active projectile list
- Level2Panel checks each enemy projectile against both players every frame
- On collision: `player.hit()` → 1 HP damage + i-frames, projectile consumed

### Files Created
- **FinalBoss.java** — Full boss class with 3-phase state machine, Phase 1 attack AI
  (fireball barrage + charge/ram), enemy projectile management, animated sprites,
  health bar, damage flash, collision detection. Takes Player1/Player2 refs for
  charge collision checking.

### Files Modified

  player refs), boss update/draw calls, player projectile→boss collision,
  enemy projectile→player collision loop

- Projectiles spawn from the left edge of the boss, move horizontally left at speed 7

- Boss slides off the right edge at 8 px/frame

- Each bolt spawns stationary at the boss's left edge, centred vertically
- After 0.5s (50 frames), each bolt calculates a direction vector toward a player
  and launches in a straight line at speed 6 — does NOT track after launch
- In multiplayer, target player chosen randomly per bolt (`rng.nextBoolean()`)
- Bolt display size: 84×42 (50% from 168×83)
- Repeats 3 times, then transitions to RETURNING

  - Two new bolts spawn at ±20° angles (up-left and down-left)
- Fork angles calculated via `Math.toRadians(20)` + `cos`/`sin` decomposition
- Bolt display size: 97×36 (50% from 193×71), base speed 7

## Loser Screen — Game Over Overlay (March 3, 2026)

### Overview
When a player's HP reaches 0, their sprite falls straight down off the bottom of the
screen (5 px/frame death-fall animation). Once ALL players have fallen off screen
(single-player: P1 dead; multiplayer: BOTH P1 AND P2 dead), the loser screen image
(`loserscreen.png`, 612×408) appears centred on the 1000×1000 panel with a dimmed
background overlay.

### Death-Fall Mechanic
- When `hp` reaches 0, `falling` flag activates in the player's `update()` method
- Normal movement and animation are skipped; player falls at `FALL_SPEED = 5` px/frame
- Once `y > panelHeight + 50`, `fallenOffScreen` becomes true
- `hasFallenOffScreen()` getter used by level panels to detect death completion

### Loser Screen Behaviour
- No stats are displayed — just the centred image with a semi-transparent black backdrop
- After a short delay (50 frames / 0.5s), `isReady()` returns true
- Player presses **SPACE** to return to the main menu (same as the score screen)
- Loser screen is mutually exclusive with the score screen — only one can trigger

### Multiplayer Rule
In multiplayer mode, the loser screen only appears when **both** Player 1 AND Player 2
have died and fallen off screen. If only one player dies, the other can continue playing.

### Files Created
- **LoserScreen.java** — Loads `loserscreen.png`, draws centred overlay with dim

## Health Packs — Floating HP Pickup (March 3, 2026)

### Overview
Health packs are floating pickup items that spawn periodically and drift left across
the screen in a sine-wave pattern. When a player touches one and has fewer than 5 HP,
they regain 1 heart.

### Sprite
- `res/New Graphics/Extra/Healthpack.png` (612×408 native)
- Displayed at **60×50**

### Spawn Rate
| Mode | Interval |
|------|----------|
| Single Player | 15 seconds (1500 frames) |
| Multiplayer | 7.5 seconds (750 frames) |

Packs spawn just off the right edge at a random Y between 50–900.

### Movement
- **Horizontal drift:** 2 px/frame leftward
- **Vertical wave:** sine wave, amplitude 40 px, period 300 frames (~3 seconds)
  `y = baseY + 40 * sin(tick * 2π / 300)`

### Pickup Rules
- AABB collision check against each player’s bounding box
- If the player’s HP is below 5 (MAX_HP), they gain 1 HP and the pack is consumed
- If the player is already at 5 HP, the pack passes through (no pickup)
- Packs are removed when consumed or when they drift off the left edge

- **HealthPack.java** — Sprite, sine-wave movement, AABB collision, consume tracking,
  static `loadSharedImage()` loader

## Bug Fix: Level Reset — Score Screen Persisting (March 3, 2026)

### Problem
After beating a level's boss and returning to the main menu, re-entering the same
level would immediately show the score/victory screen again instead of starting fresh.

### Root Cause
Level2Panel's `resetTimer()` cleared `scoreScreenTriggered = false` but did NOT null
out the `finalBoss` object. The game loop could call `updateParallax()` between
`resetTimer()` and `spawnFinalBoss()`, see the old defeated boss + the cleared flag,
and immediately re-activate the score screen.

### Fix
1. **Level2Panel.java** — Added `finalBoss = null;` to `resetTimer()` (matching
   what Level1Panel already does with `firstBoss = null`)
2. **MainWindow.java** — Moved `level1Running = true` and `level2Running = true`
   to the **end** of `showLevel1()` and `showLevel2()`, after all reset operations.
   This prevents the game loop from updating the level before it's fully initialized.
